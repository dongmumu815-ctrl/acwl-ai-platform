# 业务状态码改进方案

## 问题分析

当前API使用HTTP状态码作为业务错误码，存在以下问题：
1. 与标准HTTP状态码语义冲突（如402本意是"Payment Required"）
2. 客户端难以区分HTTP层错误和业务层错误
3. 不利于后续扩展和维护

## 改进方案

### 1. 统一响应格式

所有接口统一返回HTTP 200，通过业务状态码区分成功和失败：

```json
{
    "code": 0,
    "message": "success",
    "data": {
        "access_token": "...",
        "data_key": "...",
        "expires_in": 86400
    }
}
```

### 2. 业务状态码定义

| 业务码 | 描述                           | 原HTTP码 |
|--------|--------------------------------|----------|
| 0      | 成功                           | 200      |
| 1001   | 参数缺失或格式错误             | 400      |
| 1002   | 签名验证失败                   | 401      |
| 1003   | 请求时间戳不在允许窗口         | 402      |
| 1004   | 请求已被重放（nonce已使用）    | 403      |
| 1005   | appid无效或未授权              | 404      |
| 5000   | 内部服务器错误                 | 500      |

### 3. 错误响应格式

```json
{
    "code": 1002,
    "message": "签名验证失败",
    "data": null
}
```

### 4. 实现步骤

1. 创建统一响应模型
2. 定义业务状态码枚举
3. 修改异常处理逻辑
4. 更新接口实现
5. 修改API文档

## 优势

1. **语义清晰**：业务码专门表示业务逻辑错误
2. **易于扩展**：可以灵活添加新的业务错误码
3. **客户端友好**：统一的响应格式，便于处理
4. **标准兼容**：HTTP状态码回归其本来用途
5. **向后兼容**：可以逐步迁移，保持兼容性
